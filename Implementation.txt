- Rendered Content Warning Heuristic Patch — reduce false WARN on SPAs/canvas apps
  - Root cause: assertRenderedContent() used a single check (documentElement rect > 100x100).
    CyberChef and similar canvas-heavy/SPA apps frequently failed this check in headless mode,
    producing noisy "Document has no viewport-sized layout yet" warnings on every journey.
  - Replaced single-check logic with 3-signal heuristic — PASS if ANY signal is true:
    A) Viewport layout size: rect.width >= 200 AND rect.height >= 200
    B) Visible element signal: scan up to 1500 elements, count visible ones (display/visibility/
       opacity/size checks), PASS if visibleCount >= 1
    C) DOM density signal: PASS if body has >= 20 descendant elements (tolerant for canvas-heavy)
  - WARN (soft-fail) only when ALL 3 signals fail — true empty/broken pages still warned
  - Stabilization delay: waitForLoadState("load", 10s timeout) + 600ms render delay
    before evaluating signals (up from bare 500ms waitForTimeout)
  - Warning message improved: "Rendered content check inconclusive (rect=WxH, elements=E, visible=V)"
    with evidence values — no longer implies bot-check/consent/interstitial
  - Eliminated code duplication: smoke.ts Step 5 now calls shared assertRenderedContent()
    from assertions.ts instead of inlining identical logic
  - Executor assert case already called assertRenderedContent() — no changes needed
  - Consistent across smoke and journey execution paths
